4. AI & Scoring Logic

Tier Classification Engine

Location: server/src/engine/ (tierEngine, paceEngine)

  • Input: Last 30 runs (distance, pace, frequency)
  • Scoring weights: Performance 40%, Volume 15%, Consistency 15%, Recovery 15%, VO2max 10%, Compliance 5%
  • Output: beginner (0-39), intermediate (40-69), advanced (70-100)
  • VO2max estimation: Cooper formula from best 12-min effort or pace extrapolation
  • Age grading: Adjusts expectations based on age+gender

Kendu Economy (v2)

Location: server/src/engine/kenduEngine.ts

Earning (1 Kendu = 1 km effort)

  • Base: 1 Kendu per km (floor)
  • Coach multiplier: 1.5x (if pre-run AI check-in or club session attendance)
  • Personal Best: +5 (pace PB within 90-day window OR distance PB all-time)
  • Workout complete: +3
  • Training plan complete: +20
  • 4+ runs/week: +5
  • 7-day streak milestone: +10
  • Event attendance: +8
  • Daily cap: 50 Kendu/day (IST timezone)

Leveling Formula

Level N requires: 10 * 1.5^(N-1) lifetime Kendu earned
Level 2: 10K | Level 5: 66K | Level 10: 513K | Level 15: 3,495K

XP & Gamification

  • +25 XP per run logged
  • +50 XP per challenge completed
  • +100 XP per achievement earned
  • Level formula: 100 * level^1.5 XP per level
  • Streak tracking: consecutive days with a run (IST timezone)

AI Chat (Claude-powered)

  • Model: Claude (Anthropic API)
  • Context: Last 20 messages + user profile + recent runs + tier + goals
  • System prompt includes runner's bio, stats, tier, transformation plan
  • Daily message limit (based on subscription tier)
  • Token usage tracked in ai_usage table

5. User Flows

Registration → First Run

  1. User enters: name, phone, email, password, invite code
  2. Optional: profile photo (currently broken — not stored)
  3. Redirect to /profiling (8-step AI questionnaire)
  4. AI analyzes responses → creates runner_profile + estimates VO2max
  5. Dashboard shows "Get Started" checklist (photo, profiling, first run, first event)
  6. User taps Run button → GPS tracker starts
  7. Run completes → saved to activities → Kendu earned → analysis shown

Run Recording Flow

  1. Tap floating Run button (or navigate to /run/track)
  2. IDLE state: map centered on location, Start button
  3. RUNNING: GPS tracking (watchPosition), live distance/pace/time overlay
  4. PAUSED: Timer paused, resume/stop options
  5. FINISHED: RPE selection → Save → POST /runs/log
  6. Post-save: AI analysis card (score, tags, commentary)
  7. Then: Kendu earned breakdown modal
  8. Option: Share card to Instagram/WhatsApp

Community Creation (Admin-Gated)

  1. User submits request (name, purpose, category, contact)
  2. Admin sees pending requests with user's Kendu balance
  3. Admin approves → 1000 Kendu charged → community created + subscription started
  4. Admin rejects → no charge, user notified
  5. Monthly upkeep: 20 Kendu auto-deducted. Unpaid = dormant.

6. Known Issues & Integration Gaps

1. Profile Photo Never Stored BROKEN

Frontend collects file in registration, converts to DataURL preview, but auth.routes.ts only accepts JSON body. The profile_image_url column stays null for all users. Photo upload endpoint doesn't exist.

2. Kendu Not Earned After Run Save CRITICAL

POST /runs/log saves the run and awards XP (+25), but does NOT call calculateKenduForRun(). The /kendu/earn endpoint exists but is never triggered automatically. Frontend must call it separately (but currently doesn't after save).

3. PostRunKenduModal Has No Data BROKEN

The modal component exists and is imported in RunTrackerPage, but since Kendu earning isn't triggered, it never receives data to display.

4. No Map in Run Tracker MISSING

GPS coordinates are collected (lat, lon) during tracking but no visual map is rendered. Users can't see their route while running or after finishing.

5. Challenge Auto-Resolution Only on Page Visit DESIGN

resolveExpiredChallenges() is called when GET /challenges is hit. If nobody visits the page, expired challenges stay unresolved indefinitely. No scheduled task or cron exists.

6. Community Upkeep Never Auto-Triggers DESIGN

checkAllUpkeepDue() exists but is only called from admin panel button click. No automated schedule. Communities can avoid upkeep indefinitely if admin forgets.

7. AI Profiling Completion Not Tracked in Dashboard BUG

Dashboard checklist hardcodes profiling as `done: false`. It should query runner_profiles.profiling_complete to determine real status.

8. Strava Tokens Table Still Exists CLEANUP

Strava integration was removed and replaced with GPS tracker, but strava_tokens table and related schema remain. Dead code/schema.

9. ChatFAB + Run FAB Position Conflict UX

AppShell renders ChatFAB (floating AI chat button) at bottom-right. Adding a center Run FAB will conflict visually. Need position coordination.

10. Share Card Has No Post-Run AI Data MISSING

SharePage shows distance/pace/time but no AI rating, performance tags, or personalized commentary. The "shareable card" lacks the wow factor.

11. Invite Code Required But No Self-Serve Flow UX

Registration requires invite_code (hard block). No waitlist-to-invite pipeline. If code runs out, new users are completely blocked.

12. No Rate Limiting on API SECURITY

No express-rate-limit or similar middleware. Login, register, and AI chat endpoints are unprotected against brute force or abuse.

7. Gamification Audit

What Works

  • XP earning on run save (+25 per run)
  • Streak tracking (consecutive days)
  • Level progression formula
  • Weekly challenges generation
  • Achievement system (schema + check endpoint)
  • Kendu economy (earn/spend/balance)

What's Disconnected

  • Kendu not auto-earned — needs to be triggered in run save flow
  • Achievements not auto-checked — only triggered on explicit API call
  • No push notifications for milestones — streak milestones, level-ups don't notify
  • Challenge completion not tied to runs — challenges track "complete" boolean but don't auto-detect from run data
  • Event XP not auto-awarded — check-in exists but doesn't trigger XP/Kendu

Economy Health Concerns

  • Community creation (1000 K) takes 12+ months for casual runners — possibly too restrictive for growth
  • No Kendu seeding mechanism for new users (0 balance, cold start)
  • Upkeep deduction has no notification before it happens
  • Gift fee (15%) may discourage social economy (high friction)

8. Security Considerations

  • OK Passwords hashed with bcrypt (cost 10)
  • OK JWT-based auth with middleware protection
  • OK Input validation with Zod on critical endpoints
  • OK Helmet.js for security headers
  • OK CORS restricted to client URL
  • WARN No rate limiting on any endpoint
  • WARN No CSRF protection (JWT in header mitigates but not fully)
  • WARN SQLite file not encrypted at rest
  • WARN Profile images as base64 in DB could balloon DB size
  • WARN No input sanitization on community posts/chat (potential XSS via stored content)

9. Performance Considerations

  • OK SQLite is fast for < 10K users
  • OK React Query caching reduces API calls
  • OK Code splitting with lazy() on all pages
  • WARN No pagination on community posts (loads all)
  • WARN Kendu leaderboard queries all users (no LIMIT)
  • WARN Chat history loads all messages (no virtualization)
  • WARN GPS watchPosition with no throttle could drain battery
  • WARN Base64 images in JWT /me response would be huge

10. Recommendations for Auditor

Priority Fixes

  1. Wire Kendu earning into run save flow (most impactful — the whole economy is dead without it)
  2. Implement profile photo storage (base64 or file upload)
  3. Add rate limiting (at minimum on /auth/login, /auth/register, /chat)
  4. Auto-trigger achievement checks after runs
  5. Add scheduled upkeep processing (or at minimum, trigger on user login)

Architecture Questions

  • Should SQLite be replaced with PostgreSQL before scaling past 1K users?
  • Is base64 image storage sustainable? Consider S3/Cloudinary for profile photos.
  • Should the Kendu earning be moved server-side (in run save transaction) rather than relying on separate client call?
  • Is the AI chat token limit sufficient? What happens when context window is exceeded?
  • Should challenges and upkeep use a proper job queue (Bull, Agenda) instead of lazy evaluation?

Testing Gaps

  • No automated tests exist (unit, integration, or e2e)
  • No CI/CD pipeline beyond Railway auto-deploy
  • No error monitoring (Sentry, etc.)
  • No performance monitoring (APM)
